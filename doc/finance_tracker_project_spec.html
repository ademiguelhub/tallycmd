Tallycmd — project specification

Personal finance web app · Architecture brief · April 2026

Purpose

A personal finance web app to import bank statements incrementally, categorise and adjust transactions (including split/settled expenses), and visualise spending. Designed as a multi-user system from the start, with the primary user being the developer.

Technology stack

Solution structure

Tallycmd.Shared

Class library. No dependencies on API or UI.

  • Domain DTOs
  • Service interfaces
  • Shared utilities

Tallycmd.Api

ASP.NET Core Web API. Single source of truth for all data and auth.

  • EF Core DbContext
  • ASP.NET Identity
  • JWT issuance
  • Refresh token management
  • Domain service classes
  • REST endpoints

Tallycmd.Ui

Blazor Server app. Pure client — no DB or Identity dependencies.

  • Typed HttpClient services
  • JWT auth state provider
  • In-memory token store
  • Auth delegating handler
  • Tailwind UI components

The API and Ui projects both reference Shared. The Ui project has no direct reference to the Api project — all communication is over HTTP. This means a future mobile client, CLI import script, or alternative UI can plug into the same API without any refactoring.

Key architecture decisions

Blazor render mode

InteractiveServer, prerendering disabled

Eliminates the double-lifecycle class of bugs (JS interop not ready, operations that throw in prerender). Acceptable brief loading state for a personal tool. Reconnects automatically via SignalR.

Frontend ↔ backend communication

Typed HttpClient with a delegating handler, not direct DI service injection

Components call injected service interfaces (e.g. ITransactionService) whose implementation wraps HttpClient calls. Auth token attachment is transparent via the handler. The interface contract is decoupled from transport — a future client only needs a new implementation, not a new API.

Database layout

Single database, single DbContext inheriting from IdentityDbContext

Preserves foreign key integrity between domain entities and the user table. Enables natural LINQ joins across users and transactions. Consistent with the work project pattern which proved this trade-off worthwhile.

Authentication model

JWT access token (in-memory, scoped to Blazor circuit) + refresh token (HttpOnly cookie)

Access token never touches browser storage — protected from XSS. Refresh token in HttpOnly cookie is invisible to JavaScript and sent automatically by the browser. Silent refresh on circuit reconnect means users are never prompted to re-login within the 30-day session window.

Auth in the Blazor project

Custom JwtAuthenticationStateProvider — no ASP.NET Identity references in the Ui project

Blazor's [Authorize] and <AuthorizeView> work via the AuthenticationStateProvider abstraction. The custom implementation reads claims from the in-memory JWT, attempts a silent refresh on circuit start if no token is present, and calls NotifyAuthenticationStateChanged after login/logout.

Session lifecycle

User logs in
→
API issues JWT + sets HttpOnly refresh cookie
→
JWT stored in scoped InMemoryTokenStore
Circuit breaks / tab closes
→
JWT lost; refresh cookie survives in browser
New circuit starts
→
GetAuthenticationStateAsync finds no token
→
POST /api/auth/refresh (browser sends cookie automatically)
→
New JWT stored; user is seamlessly authenticated
Refresh token expires (30 days) or user logs out
→
Redirect to login page
